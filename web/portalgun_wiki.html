What portalgun does

Every portalgun install performs five actions in one shot:

  1. Installs the tool (apt-get install, or git clone + auto-detect compile)
  2. Records it in the registry at /var/lib/portalgun/registry/<type>/<name>.json
  3. Appends it to the v1 install scripts (Kali_Config/installable_packages.txt or Kali_Config/install_github_tools.sh) inside PORTALGUN_MANAGED_START/END markers, so clean rebuilds include it
  4. Updates the web manifest at /opt/tools-docs/portalgun_tools.json so the tools dashboard shows it
  5. Symlinks the binary to /usr/local/bin/ so it’s callable by name from any shell (Linux targets only; never shadows system commands)

The pattern: installing teaches the system. Run portalgun install… once on the master image and every future build/clone has the tool.

Installing portalgun

cd ~/portalgun
sudo ./install.sh

This:

  • Copies portalgun to /opt/portalgun/
  • Symlinks /usr/local/bin/portalgun
  • Creates /var/lib/portalgun/registry/{apt,github}/ and /var/log/portalgun/
  • Installs zsh completion to /usr/share/zsh/site-functions/_portalgun
  • Drops portalgun_tools.html and portalgun_wiki.html into /opt/tools-docs/
  • Enables portalgun-firstboot.service (runs once on first boot of clones)
  • Adds a banner link to the main tools-docs index.html

install apt

sudo portalgun install apt nmap

Result: nmap is installed, registered to /var/lib/portalgun/registry/apt/nmap.json, appended to ~/Kali_Config/installable_packages.txt (inside the managed block), and shown on the tools dashboard.

If the package is already installed, portalgun skips the install but still registers it.

install github

sudo portalgun install github <github-url> [target-dir]

Examples

sudo portalgun install github \
    https://github.com/akamai/BadSuccessor \
    /opt/tools/windows/exploit

sudo portalgun install github \
    https://github.com/tomnomnom/httprobe \
    /opt/tools/linux/recon

If the target-dir is omitted, portalgun defaults to /opt/tools/misc. Use tab completion: portalgun install github <url> /opt/tools/<TAB>.

The target directory category drives behavior:

  • /opt/tools/windows/… — assumed to be Windows tools you transfer to targets; compile is skipped, no PATH symlink created
  • /opt/tools/linux/…, /opt/tools/enumeration/…, others — compile if a build system is detected, PATH symlink created if executable lands at tool root

For install github, after cloning, portalgun inspects the repo root for these markers in priority order:

Missing toolchains are auto-installed once before building (apt install cargo, apt install golang, etc.).

Target-dir override: if the target-dir contains /windows/, compile is skipped even when a build system is detected. Windows tools usually ship pre-compiled .exes.

Tool status codes

StatusMeaning
okInstalled and (if compiled) built successfully
source_onlyClone succeeded but no build was attempted or needed
build_failedClone OK, build attempted, build failed — see /var/log/portalgun/build-<tool>.log
skippedClone OK, build skipped (windows target)

Build failures never block registration. The tool is still cloned and discoverable; you can portalgun install github <url> again after fixing build deps.

doctor

portalgun doctor

Read-only diagnostic. Reports:

  • Registry totals (apt/github counts, build_failed list)
  • /opt/tools/ audit: complete vs source-only vs empty
  • Total /usr/local/bin → /opt/tools symlinks, including broken ones
  • Symlinks that shadow system commands (warning — these are dangerous)
  • Service status: tools-server, portalgun-firstboot, bloodhound
  • Web manifest drift vs registry

To remove dangerous shadows:

sudo portalgun doctor --fix-shadows

sanitize

DESTRUCTIVE. Run only on a master image right before shutdown. Confirms with y unless --yes is passed. 
sudo portalgun sanitize

What it does:

  1. Stop BloodHound containers cleanly
  2. Clear bash/zsh history (root + all /home/* users)
  3. Truncate /var/log/* + rotate journal
  4. apt-get clean
  5. Remove /etc/sudoers.d/temp_install if master_setup.sh left it
  6. Clear DHCP leases + NetworkManager state
  7. Clear /tmp and /var/tmp
  8. fstrim -av + zero free space (qcow2 compresses small)

What sanitize does NOT remove:

  • BloodHound admin password (lives in the postgres volume) — ships in clone
  • Firefox saved passwords + bookmarks (live in the profile) — ships in clone
  • portalgun registry, /opt/tools/

These are usually what you want in the master. If not, edit them out before sanitize.

First-boot service

portalgun-firstboot.service runs once on the first boot of each clone and:

  • Regenerates /etc/machine-id (systemd, dbus, DHCP client ID)
  • Regenerates SSH host keys (/etc/ssh/ssh_host_*)
  • Clears any leftover shell history
  • Removes /var/lib/systemd/random-seed
  • Touches /var/lib/portalgun/firstboot-done and disables itself

This prevents two clones on the same network from fighting over identical machine-id or SSH fingerprints.

Services & default access

What the master image exposes after a fresh ./install.sh:

ServiceURLUsernamePassword
Tools dashboard (portalgun) http://<vm-ip>:1337 — (no auth)
BloodHound CE http://<vm-ip>:1338 admin Bloodhound123!!
BloodHound app DB (postgres) app-db:5432 (container-internal only) bloodhound bloodhoundcommunityedition
BloodHound graph DB (neo4j) graph-db:7687 (container-internal only) neo4j bloodhoundcommunityedition
The BloodHound admin password is baked into the master image (lives in the postgres seed volume). Every clone of this VM gets the same admin password until someone changes it via the UI. If you change it on a clone, it stays changed only on that clone; the next deploy of the master image will still have Bloodhound123!!.

To rotate the password in the master: log in to the UI, change it, then sudo portalgun sanitize doesn’t clear it (postgres volume is preserved). To re-seed the bloodhound volumes for distribution, re-snapshot them after changing the password — see the BloodHound section of the main README for the snapshot commands.

Manage BloodHound from the shell

sudo bloodhound-ce start          # bring up all 3 containers
sudo bloodhound-ce stop           # stop all containers
sudo bloodhound-ce status         # docker compose ps
sudo bloodhound-ce logs           # follow bloodhound logs
sudo bloodhound-ce resetpwd       # reset admin password (interactive)

VM cloning workflow

# On the master VM, after the image is fully built:
sudo portalgun doctor              # verify everything is healthy
sudo portalgun sanitize            # confirm with y
sudo shutdown -h now

# On the hypervisor host:
sudo qemu-img convert -O qcow2 -c \
    /var/lib/libvirt/images/master.qcow2 \
    ~/share/kali-$(date +%Y-%m-%d).qcow2

# Distribute. Recipients boot the qcow2 — portalgun-firstboot
# regenerates identity automatically on first boot.
Security: the seed tarballs contain real credentials (Firefox saved logins, BloodHound admin password hash). Do not redistribute the master VM externally without first replacing or removing these.

Troubleshooting

Tool not on PATH after install

  1. Was it installed to a /opt/tools/windows/… path? Windows targets are intentionally skipped (PE binaries don’t run on Linux).
  2. Does a system command with the same name exist? portalgun never shadows /usr/bin/, /bin/, /usr/sbin/, /sbin/.
  3. Run portalgun doctor — it reports total symlinks and broken ones.

Build failed for a github tool

Check /var/log/portalgun/build-<toolname>.log for the actual error. Common causes:

  • Missing build deps that auto-install didn’t catch (try apt install build-essential)
  • Rust crates that don’t compile with current rustc — drop --locked from compile_cmd
  • Old Gopkg.lock projects — modern Go won’t build them; they need go mod init conversion

"Dangerous shadow" detected by doctor

A repo created a script named find / grep / etc. that got symlinked. Run:

sudo portalgun doctor --fix-shadows

GitHub API rate limit during install

portalgun uses three tiers: hardcoded URL → API → release-page scrape. Tier 3 doesn’t hit the API, so installs work even when rate-limited. If you see "API rate limit exceeded," portalgun automatically falls through.

Registry vs web manifest drift

portalgun doctor warns if these disagree. Run any portalgun install to refresh the manifest, or manually: sudo python3 /opt/portalgun/lib/sync_web.py /var/lib/portalgun/registry /opt/tools-docs/portalgun_tools.json
