portalgun wiki

Kali Linux master image tool installer & VM lifecycle manager — comprehensive reference

Quick-Start

Get a freshly-cloned Kali VM to fully-loaded pentest machine in three steps.

1. Clone and install portalgun itself

$ git clone https://github.com/yourorg/portalgun /opt/portalgun
$ sudo /opt/portalgun/install.sh

2. Replay the master bundle (all phases)

$ portalgun install all /opt/portalgun/portalgun_bundle.json

3. (Optional) Open the Web UI

$ # Web UI starts automatically as a systemd service
$ xdg-open http://localhost:1337
Tip
The Web UI gives you real-time output streaming, per-phase control, and one-click export — ideal for interactive sessions. Use the CLI for scripting and CI pipelines.

Common first-run workflow

$ portalgun status            # verify registry is populated
$ portalgun list github        # confirm github tools registered
$ portalgun doctor             # health check

Architecture

portalgun is split into three layers that work together:

LayerComponentDescription
Web UIweb/ + data/tools_server.pyFlask/FastAPI server on :1337. Tool search, admin console, streaming install output.
CLIbin/portalgunBash/Python entrypoint. All install/update/export/lifecycle commands.
Installersinstallers/ + lib/Phase drivers (apt, github, pip, cargo), build-system detection, registry writes.

Data flow

portalgun_bundle.json
        │
        ▼
  install engine ──► apt phase   ──► dpkg + registry/apt/
                 ──► github phase ──► /opt/tools/ + symlink + registry/github/
                 ──► pip phase    ──► /opt/pentest-venv/ + registry/pip/
                 ──► cargo phase  ──► ~/.cargo/bin/ + registry/cargo/
                                              │
                                              ▼
                                 portalgun_tools.json  ◄── Web UI search

Web UI — Port 1337

portalgun ships a self-hosted web application that acts as the primary human interface for tool discovery and install management. It is served by a Python HTTP server and registered as a systemd service (tools-docs.service).

$ systemctl status tools-docs
$ systemctl restart tools-docs
Info
The UI is served on 0.0.0.0:1337. On a Proxmox VM it is accessible from the host via the VM's IP. On a bare machine it is localhost-only unless you punch a firewall hole.

Tool Search

The main page presents a full-text search bar backed by portalgun_tools.json. The index is rebuilt whenever you run portalgun register or portalgun export.

Search fieldExample
Tool namecertipy
Category tagactive-directory
Install methodgithub
Description keywordkerberos

Each result card shows the tool name, install type badge, install path, and (for GitHub tools) the source URL. Clicking a card opens the detail view with build logs and a re-install button.

Admin Console gear icon

Access the admin console via the gear icon in the top-right corner of the UI. It is the control plane for all mass-install, update, and export operations.

Install All

Replays the full portalgun_bundle.json toolset. Before starting, select which phases to run using the phase checkboxes (see Phase Selector). Phases execute in order:

1 · apt 2 · github 3 · pip 4 · cargo

The install streams live output into the Output Terminal. Each tool's result is colour-coded. The registry is updated incrementally as each tool succeeds.

Update All

Runs a full system update across all package managers and version-controlled repos:

  • apt-get upgrade — upgrades all APT packages
  • git pull — pulls latest commits for every registered GitHub tool
  • pip install --upgrade — upgrades all packages in /opt/pentest-venv
  • cargo install --force — reinstalls/upgrades cargo tools

Update output streams into the same terminal. Failed pulls (e.g. detached HEAD, upstream force-push) are highlighted in red and skipped without aborting the run.

Export Bundle

Clicking Export Bundle triggers a portalgun export server-side and serves the resulting portalgun_bundle.json as a download. Use this to:

  • Back up the current installed state to git
  • Share a curated toolset with a teammate
  • Capture the state before a destructive sanitize

Phase Selector

Four checkboxes above the install button let you limit which phases run. Useful when iterating on a single phase without waiting for the full 2000-tool install:

PhaseCheck to runTypical use
apt✓Reinstall only system packages
github✓Re-clone & rebuild custom tools
pip✓Fast test of Python tooling changes
cargo✓Rebuild Rust tools only

Unchecking all four and clicking Install All is a no-op. The CLI equivalent is --only-<phase> or --phases=.

Individual Installs

The admin console exposes four tabs for single-item installs without touching the bundle:

TabInputAction

All individual installs stream output in the terminal and append the new entry to the registry. Run portalgun export afterwards to persist the change to portalgun_bundle.json.

portalgun install apt <package>

Installs a single package via apt-get install -y and writes a registry entry under /var/lib/portalgun/registry/apt/.

$ portalgun install apt nmap
$ portalgun install apt golang-1.21

The package name is passed verbatim to apt and must exist in the configured repositories. For packages not in apt, use install github or install pip.

portalgun install github <url> [target]

Clones a GitHub (or any git) repository and auto-detects the build system to compile and install it.

$ portalgun install github https://github.com/ropnop/kerbrute
$ portalgun install github https://github.com/byt3bl33d3r/CrackMapExec cme

The optional target overrides the directory name derived from the URL. If omitted, the repo name is used.

Clone path

Tools are cloned to:

/opt/tools/<os>/<category>/<name>/source/

Where <os> is linux or windows (detected from repo metadata) and <category> is inferred from the repo topics or a default of misc.

Build system detection

portalgun scans the cloned root and picks the first matching build system:

Indicator fileBuild systemBuild command
go.modGogo build ./... && go install ./...
Cargo.tomlRust/cargocargo build --release
setup.py / pyproject.tomlPythonpip install -e . into pentest-venv
*.csproj / *.slnC# / .NETdotnet build -c Release
CMakeLists.txtCMakecmake . && make -j$(nproc)
Makefilemakemake -j$(nproc)
build.gradleGradle./gradlew build

After a successful build, produced binaries are symlinked into /usr/local/bin/ so they are immediately available on PATH without prefix.

Build log

/var/log/portalgun/build_<toolname>.log

portalgun install all [bundle.json]

Replays the full install sequence from a bundle file. If no path is given, defaults to /opt/portalgun/portalgun_bundle.json.

$ portalgun install all
$ portalgun install all ~/custom_bundle.json
$ portalgun install all --only-pip
$ portalgun install all --phases=apt,github
$ portalgun install all --phases=pip          # single phase shorthand

Flags

FlagDescription
--only-aptRun only the APT phase
--only-githubRun only the GitHub clone+build phase
--only-pipRun only the pip phase
--only-cargoRun only the cargo phase
--phases=a,b,…Run named phases in the given order
--dry-runPrint what would be installed, make no changes
--forceReinstall tools even if already present

Phase execution order

1 · apt 2 · github 3 · pip 4 · cargo

Phases always execute in this order regardless of how they appear in the bundle file. apt must complete before github (some tools need libssl-dev, libffi-dev, etc.).

Note
portalgun skips already-registered tools by default. To force a clean reinstall, pass --force or remove the tool's registry file first.

portalgun update

Performs a full system update across all four phases in a single command. Equivalent to running each update manually:

[apt]    apt-get update && apt-get upgrade -y
[github] git pull (all registered repos in /opt/tools/)
[pip]    pip install --upgrade -r <requirements>
[cargo]  cargo install --force (all registered crates)

Repos with local uncommitted changes are stashed before pulling and re-applied after. Repos that fail to pull are logged and skipped.

portalgun register

Walks the installed state of the system and backfills the registry from what is actually present — useful when the registry is out of sync (e.g. after a manual install or a restored snapshot).

$ portalgun register

Detection strategy per phase:

PhaseDetection method
aptdpkg -l cross-referenced with known package list
githubWalk /opt/tools/, detect .git directories, extract remote URL
pippip list inside /opt/pentest-venv
cargocargo install --list

Run portalgun export after register to persist the updated registry into portalgun_bundle.json.

portalgun export [outfile]

Reads all registry files and serialises them into a portalgun_bundle.json in the v2 format. If outfile is omitted, writes to the repo root.

$ portalgun export
$ portalgun export ~/backup_$(date +%Y%m%d).json

Commit the exported file to git to version-control the master toolset:

$ portalgun export
$ git add portalgun_bundle.json
$ git commit -m "toolset: add certipy-ad v4.8"

portalgun list [apt|github|pip|cargo|all]

Lists registered tools. Without an argument, defaults to all.

$ portalgun list
$ portalgun list github
$ portalgun list pip | grep impacket

portalgun doctor [--fix-shadows]

Runs a series of health checks on the install and registry state:

CheckDescription
Broken symlinksDetects symlinks in /usr/local/bin/ pointing to missing binaries
Missing registry entriesTools present on disk but absent from registry
Shadow conflictsPortalgun tool overriding a system binary with the same name

$ portalgun doctor
$ portalgun doctor --fix-shadows   # auto-resolve binary name conflicts

--fix-shadows renames conflicting portalgun-installed tools by prefixing them (e.g. python-bloodhound instead of bloodhound) to avoid masking system tools.

portalgun sanitize [--yes] destructive

Prepares the VM for cloning or snapshotting by wiping machine-specific state. See VM Lifecycle for full details.

$ portalgun sanitize           # prompts for confirmation
$ portalgun sanitize --yes    # non-interactive, use in automation
Danger
sanitize wipes bash/zsh history, SSH host keys, machine-id, DHCP leases, and log files. The VM will need to reboot (or the first-boot service will run) before SSH host keys are regenerated. Do not run on a production machine.

Bundle Format — portalgun_bundle.json

The bundle is a version 2 JSON file that describes the complete toolset. It is the source of truth for portalgun install all and the Web UI's Install All operation.

{
  "version":     "2",
  "exported_at": "2026-05-27T14:32:00Z",
  "tools": {
    "apt": [
      "nmap",
      "curl",
      "golang-go"
    ],
    "github": [
      {
        "url":    "https://github.com/ly4k/Certipy",
        "target": "certipy-ad"
      },
      {
        "url":    "https://github.com/ropnop/kerbrute",
        "target": "kerbrute"
      }
    ],
    "pip": [
      "impacket",
      "certipy-ad",
      "bloodhound"
    ],
    "cargo": [
      "rustscan"
    ]
  }
}
FieldTypeDescription
versionstringAlways "2" for the current format
exported_atISO 8601Timestamp of the last portalgun export
tools.aptstring[]APT package names
tools.github[].urlstringFull HTTPS clone URL
tools.github[].targetstringOverride directory/tool name (optional)
tools.pipstring[]PyPI package names (or git+https://…)
tools.cargostring[]crates.io crate names

Editing the Bundle

The bundle is plain JSON — edit it directly in your editor or via portalgun export after installing new tools.

Workflow: add a new GitHub tool to the master set

$ # 1. Install and verify the tool works
$ portalgun install github https://github.com/example/newtool

$ # 2. Regenerate the bundle from the updated registry
$ portalgun export

$ # 3. Commit to git
$ git add portalgun_bundle.json
$ git commit -m "toolset: add newtool"

Workflow: remove a tool

$ # Remove from registry + uninstall
$ apt-get remove -y badtool
$ rm /var/lib/portalgun/registry/apt/badtool.json

$ # Regenerate bundle
$ portalgun export && git add portalgun_bundle.json && git commit -m "toolset: remove badtool"

Installation Engine — APT Phase

The APT phase installs system packages in batches to minimise round-trips to dpkg.

  • Splits package list into chunks of 50 for parallel apt invocations
  • Uses DEBIAN_FRONTEND=noninteractive for unattended operation
  • Tracks install success via dpkg -l post-install verification
  • On failure, logs the apt error and continues with remaining packages
  • Each successfully installed package gets a registry entry at /var/lib/portalgun/registry/apt/<name>.json
[apt] installing batch 1/21 (50 packages)…
[apt] nmap ✓
[apt] masscan ✓
[apt] some-nonexistent-pkg ✗ (E: Unable to locate package)
[apt] batch 1 done — 49/50 succeeded

Installation Engine — GitHub Phase

The GitHub phase handles the full lifecycle from clone to symlink. Tools are cloned to:

/opt/tools/<os>/<category>/<name>/source/

Post-build symlink logic

After a successful build, portalgun locates produced executables (ELF binaries, shell scripts, Python entrypoints) and creates symlinks in /usr/local/bin/:

/usr/local/bin/kerbrute → /opt/tools/linux/ad/kerbrute/source/kerbrute_linux_amd64
/usr/local/bin/certipy  → /opt/tools/linux/ad/certipy-ad/source/certipy

If a tool produces multiple binaries, all are symlinked. Use portalgun doctor to detect broken or shadowing symlinks.

Installation Engine — pip Phase

All pip packages are installed into an isolated virtual environment at /opt/pentest-venv rather than the system Python. This prevents version conflicts and keeps the root filesystem clean.

$ # Equivalent of what portalgun does internally
$ /opt/pentest-venv/bin/pip install --upgrade impacket certipy-ad bloodhound

Installation Engine — cargo Phase

Rust tools are installed via cargo install into the Rust toolchain's default binary path (~/.cargo/bin/ for the install user, or /usr/local/cargo/bin/ if cargo is installed system-wide).

$ # Equivalent
$ cargo install rustscan

The cargo phase is the smallest by default (1 tool: rustscan). Add tools by editing tools.cargo in the bundle and running portalgun install all --only-cargo.

Python Tools — Pentest Venv

All pip-installed tools live in an isolated virtual environment at /opt/pentest-venv. This is the canonical Python environment for offensive tooling on a portalgun-managed machine.

/opt/pentest-venv/
├── bin/
│   ├── python3  (symlink to CPython)
│   ├── pip3
│   ├── impacket-*   (many entrypoints)
│   ├── certipy
│   ├── bloodhound
│   └── …
├── lib/
│   └── python3.x/site-packages/
└── pyvenv.cfg

PATH and sudo Access

portalgun installs a profile drop-in so that /opt/pentest-venv/bin is on PATH for all users and all shells:

/etc/profile.d/pentest-venv.sh
# Contents
export PATH="/opt/pentest-venv/bin:$PATH"

For sudo invocations, the secure_path in /etc/sudoers (or a drop-in under /etc/sudoers.d/) is extended so that sudo impacket-secretsdump resolves correctly without needing the full path:

# /etc/sudoers.d/pentest-venv
Defaults secure_path="/opt/pentest-venv/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin"
Tip
If a pip tool is not found after install, source /etc/profile.d/pentest-venv.sh or open a new shell. Changes to profile.d apply to new login shells, not the current session.

VM Lifecycle — Sanitize & Clone

portalgun sanitize prepares a VM for Proxmox snapshotting or full cloning. Run it on the golden master after confirming all tools install correctly and before committing the snapshot.

What sanitize wipes

ArtefactAction
/etc/machine-idTruncated to empty (regenerated on first boot)
/var/lib/dbus/machine-idSymlinked to /etc/machine-id or wiped
/etc/ssh/ssh_host_*Deleted (regenerated by first-boot service)
~/.bash_history, ~/.zsh_historyWiped
/var/log/ (selected logs)Rotated and truncated
/var/lib/dhcp/DHCP leases cleared
/tmp/, /var/tmp/Cleared
root/user authorized_keysPreserved (intentional — re-add your key)

Recommended pre-clone checklist

[ ] portalgun install all   — full toolset installed & verified
[ ] portalgun doctor        — no broken symlinks or missing entries
[ ] portalgun export        — bundle committed to git
[ ] portalgun sanitize --yes
[ ] poweroff VM
[ ] snapshot / clone in Proxmox

First-Boot Service

A oneshot systemd service (portalgun-firstboot.service) runs on the first boot of a cloned VM and regenerates machine-specific state:

  • Generates a new /etc/machine-id via dbus-uuidgen
  • Regenerates SSH host keys via dpkg-reconfigure openssh-server
  • Disables itself after successful run so it never fires again
# systemd unit (abbreviated)
[Unit]
Description=portalgun first-boot initialisation
After=network.target
ConditionPathExists=!/var/lib/portalgun/firstboot.done

[Service]
Type=oneshot
ExecStart=/opt/portalgun/lib/firstboot.sh
ExecStartPost=/usr/bin/touch /var/lib/portalgun/firstboot.done

[Install]
WantedBy=multi-user.target

├── github/

│   ├── kerbrute.json
│   └── …  (186 files)
├── pip/
│   ├── impacket.json

Registry entry format (GitHub example)

{
  "name":        "certipy-ad",
  "type":        "github",

  "target":      "certipy-ad",

}

Rebuilding the registry

$ portalgun register   # scan installed state → write registry
$ portalgun export     # registry → portalgun_bundle.json

Web Manifest

The Web UI's search feature is backed by a JSON manifest that is separate from the per-tool registry files:

/opt/tools-docs/portalgun_tools.json

This file is regenerated by portalgun register and portalgun export. It is a flat array of tool objects designed for fast client-side fuzzy search:

[
  {
    "name":     "certipy",
    "type":     "github",
    "tags":     ["active-directory", "certificates", "python"],
    "path":     "/opt/tools/linux/ad/certipy-ad/source",
    "bin":      "/usr/local/bin/certipy"
  }
]

If the Web UI search returns stale results, regenerate the manifest:

$ portalgun register && systemctl restart tools-docs

Troubleshooting

Web UI not loading on :1337

$ systemctl status tools-docs
$ journalctl -u tools-docs --since "10 min ago"
$ systemctl restart tools-docs

Check that port 1337 is not already in use:

$ ss -tlnp | grep 1337

Tool not found after install
